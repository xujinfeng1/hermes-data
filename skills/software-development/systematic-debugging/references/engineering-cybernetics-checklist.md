# 工程控制论开发方法论

钱学森《工程控制论》核心原则，适用于所有开发任务。

## 十一项原则（每次改动前对照）

| # | 原则 | 含义 | 本次违反示例 |
|---|------|------|-------------|
| 1 | 系统观 | 先界定边界，理解整体 | 视频卡住没查 ComfyUI 状态就改代码 |
| 2 | 反馈控制 | 闭环验证，每次改完立即验证 | 连加 8 个功能，没等验证 |
| 3 | 稳定性第一 | 可回滚，基准功能不能坏 | 图片生成坏了没发现 |
| 4 | 最优化 | 先跑通再跑快 | 视频用轮询了但没先确认帧生成正常 |
| 5 | 可靠性 | 假定会出错 | 没考虑 ComfyUI 内存碎片化会卡死 |
| 6 | 信息论 | 不臆测，查日志/数据 | 凭感觉改代码，没看 actual error |
| 7 | 鲁棒性 | 为最坏情况设计 | 没处理 ComfyUI 超时降级 |
| 8 | 系统辨识 | 先诊断后开方 | 直接改代码，没先定位根因 |
| 9 | 自适应 | 不犯两次同样的错 | 中文翻译修了 4 次，因为每次只修表面（端点），没根治到引擎内部 |
| 10 | 分级递阶 | 战略→战术→操作，分层次决策 | 跳过了"先确认基础功能"这一步 |
| 11 | 人机协同 | 人在回路，关键决策问用户 | 改 UI 前没问用户 |

## 开发前检查清单（每次动手前必过）

```
□ 1. 当前系统健康吗？(health check / system_stats)
□ 2. 基准功能正常吗？(端到端一次完整测试)
□ 3. 上一步改动验证了吗？
□ 4. 这次改动范围是什么？(最小化，不顺便改别的)
□ 5. 改动后怎么验证？(具体的测试步骤)
□ 6. 失败回滚路径是什么？
```

## 常见违反信号

- "我先加这个功能，等会再测" → 违反 2/4
- "顺手把这个也改了" → 违反 3
- "应该是 X 的问题，我改一下试试" → 违反 6/8
- 连加 3+ 个功能没验证 → 违反 2/3/8
- 用户反馈问题，直接改代码不看日志 → 违反 8
