# 工程控制论开发准则

钱学森《工程控制论》衍生开发纪律。违反任一条都会导致浪费时间和用户挫折。

## 核心原则

### 1. 系统观 — 先界定边界再动手
功能不是孤立的。动手前确认：
- 这个功能依赖什么？（ComfyUI 进程？网络？模型？）
- 影响什么别的功能？（改了 A 会不会破坏 B？）
- 行业标准是什么？（Leonardo/Runway/Kling 怎么做？）

**反例：** 视频卡住 → 连续改了 8 次代码，没查 ComfyUI 状态（根因是 MPS 内存碎片）

### 2. 反馈控制 — 闭环验证每一步
每做完一个变更，必须验证：
- `node -e "new Function(js)"` 验证 JS 语法
- `curl /health` 验证服务状态
- `curl /queue` 验证工作流提交
- 实际跑一次看结果

**反例：** patch 了 4 次 HTML，没跑语法检查 → 页面按钮全部失效

### 3. 稳定性第一 — 先跑通再优化
基本功能不稳定时，不要加新功能。

**优先级：**
1. 验证当前功能正常
2. 跑通全链路
3. 再加新功能

**反例：** ComfyUI 生图还没验证稳，就开始加视频/图生图/控制台

### 4. 最优化 — 先跑通再跑快
功能没跑通前不要优化参数、UI、性能。

### 5. 可靠性 — 假定每一步会出错
关键步骤必须加 backstop：
- 翻译逻辑放在引擎函数内部（不在端点层）
- API 调用加 try/except + 日志
- 下载失败要有回退方案

### 6. 信息论 — 不臆测，先诊断
问题出现时，先查状态再下结论：
```bash
curl /health          # 服务在吗？
curl /queue           # 队列卡了？
tail logs/*.log       # 有什么错误？
ps aux | grep PROc    # 进程在吗？
```

### 7. 鲁棒性 — 为最坏情况设计
- 中文网络：HF 被墙 → 准备 mirror list
- 终端不走 VPN → 浏览器下载方案
- 模型太大 → 量化版本

### 8. 系统辨识 — 先诊断根因再开方
不要反复 patch 同一个问题。如果修了两次还没好，停下来诊断。

### 9. 自适应 — 不犯两次错
同类型错误出现第二次时，检查流程是否有系统性问题。

**反例：** 中文翻译问题出现了 3 次——翻译放在端点层，静默失败。根治：翻译移入引擎函数内部。

### 10. 分级递阶 — 战略/战术/操作分离
- 战略层：先定架构，别急着写代码
- 战术层：选型时查行业标准
- 操作层：执行时验证每一步

### 11. 人机协同 — 人在回路
耗时操作（下载 27GB 模型）让用户参与决策，别自作主张全自动跑。

## 常见违反模式

| 违反 | 症状 | 根治 |
|------|------|------|
| 不诊断就改代码 | 反复 patch 无效 | 先 curl + ps + log |
| 不加验证 | 页面崩溃发现不了 | `node -e "new Function(js)"` |
| 端点层翻译 | 静默失败无日志 | 翻译移到引擎函数 |
| 依赖没查 | 功能卡住不知道原因 | 先查依赖（ComfyUI 状态）|
